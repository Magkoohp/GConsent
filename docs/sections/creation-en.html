Methodology back to ToC

The creation of the ontology followed an iterative model consisting of the following steps: 1. Gather information from GDPR, articles, academic papers, communications from various supervisory bodies and regulatory authorities (such as the Data Protection Commissioner's office) 2. Quantify information as a set of requirements 3. Create models for representation of information 4. Test the model using- - Various use-cases for expressibility - Competency Questions for effectiveness - OntoClean for completeness - SPARQL queries for usability 5. Generate OWL2 ontology for the model 6. Ensure ontology quality using best practices and guidelines PrOnto [ref] describes the Methodology for building Legal Ontology (MeLOn), which is a 10 step process geared towards involving legal experts in the ontology building process. GConsent plans to integrate this methodology to further expand the ontology to other areas of legal compliance. Noted below are the steps or objectives common between methodologies used by PrOnto and GConsent from the MeLOn list: * Start with the aim/goal. * Identify parameters for evaluation. For GConsent, these were - coherence, efficiency, effectiveness, usability, and extendability. * State of the art survey. * Use of Graffoo. * Use of OntoClean to improve the ontology. * Use of SPARQL to test applicability.

  • Recital 32: Under Recital 32, the requirement of unambiguous ness can be satisfied, for instance, by a box being ticked, whereas an opt out model is expressly ruled out. An audit of the respective terms and conditions ought to be performed regarding pre formulated con sents (Recital 42).

  • According to Recital 32, consent should cover all pro cessing activities carried out for the same purpose or purposes. When the processing has multiple purpos es, consent should be given for all of them.

  • Article 4 (11) GDPR requires that consent be given by a freely given, informed and unambigu ous indication of the data subject's agreement to the processing of his or her personal data. In order to be informed, the consent must also comprise the pur pose of the data processing. Consent must be explic itlygiven in the event that sensitive data are processed or that data are processed for profiling purposes.

Competency Questions

The ontology is validated using the following competency questions. These questions are based on the various requirements of consent as specified under GDPR, as well as to explore the role of various entities and agents in the state of consent. The questions are also useful to validate the data instances created using the ontology.

ID Question Concepts Properties Comment
Questions about consent
C1 Who is the consent about? DataSubject isConsentForDataSubject Specifies the Data Subject.
C2 What type of Personal Data are associated with the Consent? PersonalData forPersonalData Relates to the set of Personal Data.
C3 What type of Purposes are associated with the Consent? Purpose forPurpose Specifies the set of Purposes.
C4 What type of Processing are associated with the Consent? Processing forProcessing Specifies the set of Processing (actions or activities).
C5 What is the Status of Consent? Status hasStatus Specifies the status of consent.
C6 Is the current status valid for processing? StatusValidForProcessing, StatusInvalidForProcessing hasStatus Specifies whether the current status is valid for use in processing.
C7 Who is the consent given to? prov:Person, DataController isProvidedTo Specifies the entity that was provided consent.
Questions about how the consent was created/given/acquired/changed/invalidated
P1 Who created/gave/acquired/invalidated the consent? DataSubject, Delegation isProvidedBy Specifies the Data Subject or Delegation that gave consent.
P2 If consent was created/gave/acquired/invalidated through Delegation, who acted as the Delegate? prov:Person, Delegation prov:agent Specifies the entity that provided consent in a delegation.
P3 If consent was created/gave/acquired/invalidated through Delegation, what was the role played by Delegate? prov:Role prov:hadRole Specifies the role played by the entity that gave consent in a delegation.
P4 If consent was created/gave/acquired/invalidated through Delegation, how was the delegation executed? prov:Activity prov:hadActivity Specifies the activity that provided consent in a delegation.

T1 What is the location of associated with consent? prov:Location atLocation Specifies the location where consent was created/generated/given/collected.
T2 What is the medium associated with consent? Medium inMedium Specifies the medium through which consent was created/generated/given/collected.

D1 Is the purpose or processing associated with a third party? Association, ThirdParty hasAssociation, prov:agent Specifies any third parties associated with the purposes or processing for the consent.

Ensuring Ontology Quality

The ontology follows best practices guidelines encouraged and adopted by the semantic web community. The list below lists these, along with a few comments.

  • Best Practice Recipes for Publishing RDF Vocabularies - W3C Working Group Note 28 August 2008
    These guidelines form the core of best practices, and relate to creation and usage of namespaces and content negotiation. GConsent uses a consistent naming convention based on these guidelines, and provides content negotiation in the serialisation formats listed at the top of this documentation.
  • Live OWL Documentation Environment (LODE)
    LODE is a well-known service within the semantic web community for self-documentation of ontologies. GConsent provides relevant metadata to document the ontology in a LODE-compatible manner.
  • OntOlogy Pitfall Scanner!
    OOPS! is another well-known service that detects some of the most common pitfalls appearing when developing ontologies. GConsent was tested iteratively throughout its development using the OOPS! service. Note: the latest version of GConsent shows some pitfalls in the OOPS! report, which result from its usage of external concepts and properties (mostly from PROV-0). The pitfalls can be safely discarded as there are no errors directly related to the concepts and relationships definde within GConsent.
  • WIzard for DOCumenting Ontologies (WIDOCO)
    WIDOCO extends the LODE and OOPS! services along with Licensius, WebVOWL, and Bubastis to create a set of documentation based on best practices and guidelines, that can be edited and served in a responsible manner using pre-configured files for Apache. GConsent uses WIDOCO to generate the documentation pages, and follows its set of best practices - A checklist for complete vocabulary metadata - 25 April 2017.
