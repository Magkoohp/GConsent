Motivation

The General Data Protection Regulation (GDPR) [1] is the current European data protection law enforced from 25th May 2018. It affects any service or organisation that uses personal data, and has large fines to deter non-compliance. Consent is one of the legal basis for using personal data under the GDPR (Art.6). Consent is considered valid only when it is freely given, specific, informed, and unambiguous (Art.2-11); and in the case of minors to be given by their legal guardian (Art.8). GDPR also provides rights regarding changing and withdrawing consent at any time. To demonstrate compliance with these obligations, Data Controllers should maintain a demonstrable proof of the given consent [2] by collecting and storing information on how consent was collected, used, and invalidated [3].

Semantic Web technologies are ideal for representing this information because of its flexibility in expressing concepts and relationships, e.g. using the Web Ontology Language - OWL2, using a standardised model for data - i.e. the Resource Description Framework (RDF). Existing work [4,5,6,7,8,9,10,11] has demonstrated the feasibility of using semantic web technologies for representing and querying metadata for assisting with the GDPR compliance process. Our previous work presented the Consent and Data Management Model (CDMM) [5] as a framework for information related to GDPR and its compliance using semantic web technologies, which included a preliminary OWL2 ontology for representing consent.

The work mentioned above, as well as that in the related work section features concepts mostly focused on the 'given' aspect of consent i.e. consent provided by the data subject. There is a lack of work regarding other states of consent, as well as creating a comprehensive information model around consent. Furthermore, consent as a concept is encompassing of the notion of given consent, as it also includes other aspects, or states, of consent such as "not given" or "refused" or "withdrawn" which cannot be modelled in the same manner as "given consent". This provided the motivation to create a more generic ontology for the modelling of consent states, in the form of GConsent.

Aims, Scope, and Limitations

Aims

  1. To model information associated with Consent as per the GDPR
  2. Represent consent as an entity with various states (e.g. unknown, given, withdrawn)
  3. Investigate ways to model related information (e.g. provenance, associations) using existing vocabularies
  4. To provide an OWL2-DL ontology with concepts and relationships based on above
  5. Create a list of competency questions related to consent and its assessment

Scope

  • Consent as a concept has broad definitions and applications. We limit the scope in GConsent to consent as defined within the context of the GDPR.
  • GConsent models information which will be of assistance to the process of compliance verification and demonstration. GDPR Compliance is not the goal for GConsent.
  • GConsent captures only the directly (or sometimes closely) related information regarding consent. For example, it does not capture things such as how the purpose or processing specified in the process is actually carried out.

Limitations

  • GConsent does not model the consent as a 'policy' or 'contract', and therefore does not contain more specific information such as where the data will be collected from or any temporal aspects regarding processes. This limitation can be addressed by using relevant concepts and properties from external vocabularies such as PROV-O.
  • GConsent is related to, but does not model the provenance of lifecycles for consent. GDPRov is a provenance vocabulary that can be used to model the provenance of lifecycles.
  • There is no information on how existing schemas and vocabularies can be integrated into GConsent. This limitation is used as the basis for future work in this direction, particularly the investigation of using R2RML for mapping databases containing consent into ontological representations.
  • GConsent has a role in compliance verification and demonstration, which is currently not explained in detail. This is also the basis for future work by creating a platform that provides design patterns and tools for verifying the validity of consent, and for producing documentation based on it.
