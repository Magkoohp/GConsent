Actionc back to ToC or Class ToC

IRI: https://w3id.org/GConsent#Action

`Action` is defined as an activity that uses data in a particular or specific way. The purpose of action is to specify what is happening to/with the data. For GDPR, there are four types of actions: *Collect*, *Store*, *Use*, and *Share*. To represent these, `Action` has been subclassed for each type of action as - [`CollectAction`](#CollectAction), [`StoreAction`](#StoreAction), [`UseAction`](#UseAction), and [`ShareAction`](#ShareAction). Each of these have a default instance to provide a generic target to refer to the action - [`CollectionOfPersonalData`](#CollectionOfPersonalData), [`StorageOfPersonalData`](#StorageOfPersonalData), [`UsageOfPersonalData`](#UsageOfPersonalData), and [`SharingOfPersonalData`](#SharingOfPersonalData). `Action` has subclasses because different use-cases might want to differentiate between different types of the same action. For example, different types of collection mechanisms, or different types of sharing mechanisms (such as sharing with different entities). These are possible by either further subclassing the respective actions, or creating instances. If these actions were declared as instances from the start, then the ability to further distinguish these would be lost.
Example
@prefix foaf: <http://xmlns.com/foaf/0.1/> .
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:HJP a gconsent:DataSubject ;
    foaf:name "Harshvardhan" .
:Advertising a gconsent:Purpose ;
    gconsent:hasAction gconsent:CollectionOfPersonalData ;  # <----
    gconsent:hasAction gconsent:StorageOfPersonalData ;     # <----
    gconsent:hasAction gconsent:UsageOfPersonalData ;       # <----
    gconsent:hasAction gconsent:SharingOfPersonalData .     # <----
:Marketing a gconsent:Purpose .
    gconsent:hasAction gconsent:CollectionOfPersonalData ;  # <----
    gconsent:hasAction gconsent:UsageOfPersonalData .       # <----
:Consent_hjp a gconsent:Consent ;
    gconsent:isConsentFor :HJP ;
    gconsent:givenForPurpose :Advertising ;
    gconsent:givenForPurpose :Marketing .
has sub-classes
CollectAction c, ShareAction c, StoreAction c, UseAction c
is in domain of
isActionForPurpose op
is in range of
hasAction op
has members
CollectionOfPersonalData ni, SharingOfPersonalData ni, StorageOfPersonalData ni, UsageOfPersonalData ni

Consentc back to ToC or Class ToC

IRI: https://w3id.org/GConsent#Consent

As per Article 4(11) of the GDPR, ‘consent’ of the data subject means any freely given, specific, informed and unambiguous indication of the data subject’s wishes by which he or she, by a statement or by a clear affirmative action, signifies agreement to the processing of personal data relating to him or her; In the case of this ontology, 'Consent' is a concept as well as an entity. To separate this distinction with relation to the data subject, the `Consent` class represents the consent of the data subject in its entirely, including any history and annotations for it. The class `ConsentInstance` represents the given consent at a specific time, which is linked to the Consent class. To link the `Consent` instance with the `DataSubject` instance, use the [`isConsentFor`](#isConsentFor) and [`hasConsent`](#hasConsent) properties.
Example
@prefix foaf: <http://xmlns.com/foaf/0.1/> .
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:HJP a gconsent:DataSubject ;
    foaf:name "Harshvardhan" .
:HJP_Consent a gconsent:Consent ;  # <----
    gconsent:isConsentFor :HJP .
is in domain of
givenForPurpose op, hasConsentInstance op, hasStatus op, isConsentFor op
is in range of
hasConsent op, isConsentInstanceOf op, isPurposeForConsent op

ConsentInstancec back to ToC or Class ToC

IRI: https://w3id.org/GConsent#ConsentInstance

`Consent` as an entity exists as an interface (or representation) specifying the specific consent (as permission) between the data subject and the purpose (which contains action(s) and personal data item(s)). This _entity_ stays fixed (or constant) throughout its lifetime, and is associated with various _instances_ of given consent. To represent these instances, the class `ConsentInstance` is used. `ConsentInstance` is associated with a `Consent` instance using the [`hasConsentInstance`](#hasConsentInstance) and [`isConsentInstanceOf`](#isConsentInstanceOf) properties. A consent instance defines one specific given consent. The data subject has the right to change or modify or withdraw this consent at a later date. To link these two consent instances together, the properties [`wasReplacedByConsentInstance`](#wasReplacedByConsentInstance) and [`replacesConsentInstance`](#replacesConsentInstance) are used. Attributes describing the context of specific consent instances include metadata describing specific instances of consent - such as timestamp, location, or medium. A parent property is useful to represent such attributes collectively, and to provide a common way to link these to consent instances. These are defined as [`isContextForConsentInstance`](#isContextForConsentInstance) and [`hasContext`](#hasContext) properties. A Consent Instance represents a specific status of consent applicable to the instant it defines. For e.g. a consent given at a particular instant specifies "explicit consent" to the specified purpose. Whereas, another consent given later "withdraws" this consent for the specified purpose. To capture this chain or log or provenance of change in consent, the [`changesConsentStatusTo`](#changesConsentStatusTo) property is used.
Example
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:HJP a gconsent:DataSubject ;
    foaf:name "Harshvardhan" .

:consent_hjp_1808 a gconsent:ConsentInstance ;  # <----
    rdfs:label "Email op-out" ;
    gconsent:isConsentInstanceOf :Consent_hjp ;
    gconsent:wasGivenBy :HJP ;
    gconsent:changesConsentStatusTo gconsent:ConsentStatusWithdrawn ;
    gconsent:wasGivenAtInstant "2099-12-31-00-00-00" ;

:Consent_hjp a gconsent:Consent ;
    gconsent:isConsentFor :HJP ;
    gconsent:hasConsentInstance :consent_hjp_1808 ;
    gconsent:hasStatus gconsent:ConsentStatusWithdrawn .
is in domain of
changesConsentStatusTo op, hasContext, hasExpiry, isConsentInstanceOf op, replacesConsentInstance op, wasGivenAtInstant, wasGivenAtLocation, wasGivenBy, wasGivenThroughMedium, wasReplacedByConsentInstance op
is in range of
hasConsentInstance op, isContextForConsentInstance op, replacesConsentInstance op, wasReplacedByConsentInstance op

DataSubjectc back to ToC or Class ToC

IRI: https://w3id.org/GConsent#DataSubject

Data Subject is defined as an identified or identifiable natural person (‘data subject’); an identifiable natural person is one who can be identified, directly or indirectly, in particular by reference to an identifier such as a name, an identification number, location data, an online identifier or to one or more factors specific to the physical, physiological, genetic, mental, economic, cultural or social identity of that natural person; - according to Article 4(1) of the GDPR pertaining to the definition of Personal Data. In the case of this ontology, a Data Subject refers to the person(s) as per the definition of the GDPR. Since a Data Subject is a Person, it is defined as a subclass of `foaf:Person`. The Data Subject is linked to the `Consent` instance via the [`isConsentFor`](#isConsentFor) and [`hasConsent`](#hasConsent) properties.

Delegationc back to ToC or Class ToC

IRI: https://w3id.org/GConsent#ConsentDelegation

A Delegation of consent occurs when another entity, whether a data subject or their representative or an automated mechanism, provides the consent on behalf of the intended Data Subject. The delegated consent is provided through the mechanism of a _delegation_. This involves three entities - the Data Subject the delegation occurs for, the entity that acts as the delegate, and the mechanism used to verify the authority to delegate. They are provided respectively by the properties [`isConsentDelegationFor`](#isConsentDelegationFor), [`hasConsentGivenByDelegate`](#hasConsentGivenByDelegate), and [`hasVerificationProcessForDelegation`](#hasVerificationProcessForDelegation).
Example
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:consent_hjp_1808 a gconsent:ConsentInstance ;
    rdfs:label "Emergency consent for Mr. Harshvardhan" ;
    rdfs:comment "Harshvardhan had an accident and was brought to the hospital. During this time, his vitals and other information were taken for hospital records." ;
    gconsent:isConsentInstanceOf :Consent_hjp ;
    gconsent:wasGivenBy :delegatedconsent ;
    gconsent:wasGivenAtInstant "2099-12-31-00-00-00" ;
    gconsent:wasGivenAtLocation "Hospital" ;
    gconsent:changesConsentStatusTo gconsent:ConsentStatusExplicitlyGiven .

:delegatedconsent a gconsent:ConsentDelegation ;   # <----
    rdfs:label "Delegated consent for Mr. Harshvardhan by Hospital Staff" ;
    gconsent:hasConsentGivenByDelegate "Hospital Staff Member 4826719" ;
    gconsent:isConsentDelegationFor :HJP ;
    gconsent:hasVerificationProcessForDelegation "Recorded under medical registry for emergencies" .

MinorDataSubjectc back to ToC or Class ToC

IRI: https://w3id.org/GConsent#MinorDataSubject

A Minor is a Data Subject who cannot provide their own consent (that is legally valid). Therefore, their consent must be obtained via a Guardian or Legal Representative or Parent. Minor is a subclass of Data Subject. Consent given in lieu of a Minor (by their Representative) is associated with the [`ConsentInstance`](#ConsentInstance) class/instance using the [`wasGivenBy`](#wasGivenBy) property. The ontology currently does not capture this relationship between the Minor and their Representative. This is due to required clarifications regarding how this role should be represented. For example, the `DataSubject` cannot be subclassed to create a `Representative` class, as this would make the `DataSubject` a permanent representative for the Minor. Instead, the role of 'acting as representative' is only for the instance of the given consent. Therefore, this could be represented with a `ConsentRepresentativeRole` class that links to the `MinorDataSubject` with `roleHadDataSubject` property, and the `DataSubject` that gave the consent via the `roleHadRepresentative` property. The relationship or reason or situation of this given consent can be captured by further annotating the `ConsentRepresentativeRole` instance. This approach can also be extended to other similar situations where one `DataSubject` gives consent in lieu of another. Note: A Minor can provide their own consent in certain cases, or conversely, a minor's consent may need to be provided by a 'parental' representative in some cases.
Example
@prefix foaf: <http://xmlns.com/foaf/0.1/> .
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:Child a gconsent:MinorDataSubject ;  # <----
    foaf:name "Harshvardhan Jr."
    foaf:age 10 .
:HJP a gconsent:DataSubject ;
    foaf:name "Harshvardhan" ;
:Consent_10C a gconsent:Consent ;
    gconsent:isConsentFor :Child ;
    gconsent:wasGivenBy :HJP .
has super-classes
DataSubject c

PersonalDatac back to ToC or Class ToC

IRI: https://w3id.org/GConsent#PersonalData

As per Article 4(1) of the GDPR, ‘personal data’ means any information relating to an identified or identifiable natural person (‘data subject’); `PersonalData` is associated with the `Consent` via the `Purpose` associated with that consent. This is because each purpose may have different actions over personal data, and this separation allows distinguishing such actions over the specified personal data instances. To link the `PersonalData` instance with the `Purpose` instance, use the [`hasPersonalData`](#hasPersonalData) and [`isPersonalDataForPurpose`](#isPersonalDataForPurpose) properties.

Purposec back to ToC or Class ToC

IRI: https://w3id.org/GConsent#Purpose

The model of consent is such that `Purpose` is defined as the set of `Actions` on a `Data` with an intended 'purpose'. Therefore, the same set of actions on the same set of data can have different 'purposes'. For example, consider the set of _(collect and use personal data)_ and the different purposes of _Advertising_ and _Marketing_. Since the consent for these two is separate or can be given and modified and withdrawn independently by the data subject, their modeling and representation should also reflect this separation. Therefore, the modeling is done as follows: Each consent has only one purpose or more purposes The applicable purpose then is the union of all purposes For e.g. _Advertising_ and _Marketing_ are two purposes, but if they are associated with a single consent instance, then the consent is deemed to be given for the _union of Advertising and Marketing_. Each purpose can have multiple actions and multiple data The applicable consent is for the union of actions on the union of data That is, each action is permitted over each item of data For a different purpose, a separate consent object must be created. `Purpose` is linked to `Consent` via the [`givenForPurpose`](#givenForPurpose) and [`isPurposeForConsent`](#isPurposeForConsent) properties.
Example
@prefix foaf: <http://xmlns.com/foaf/0.1/> .
@prefix gconsent: <https://w3id.org/GConsent#> .
@prefix : <http://example.com/> .

:HJP a gconsent:DataSubject ;
    foaf:name "Harshvardhan" .
:Advertising a gconsent:Purpose ;  # <----
    gconsent:hasAction gconsent:CollectionOfPersonalData ;
    gconsent:hasAction gconsent:StorageOfPersonalData ;
    gconsent:hasAction gconsent:UsageOfPersonalData ;
    gconsent:hasAction gconsent:SharingOfPersonalData .
:Marketing a gconsent:Purpose .    # <----
    gconsent:hasAction gconsent:CollectionOfPersonalData ;
    gconsent:hasAction gconsent:UsageOfPersonalData .
:Consent_hjp a gconsent:Consent ;
    gconsent:isConsentFor :HJP ;
    gconsent:givenForPurpose :Advertising ;
    gconsent:givenForPurpose :Marketing .
is in domain of
